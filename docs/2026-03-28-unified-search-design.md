# Unified Brain Search — Design Document

## Motivation

AgentLink's Feed already surfaces Meeting Recaps, Daily Briefings, DevOps items, and Project knowledge — all pre-generated by Brain. But users often need to answer: *"Who mentioned X?"*, *"Where was that discussion about Y?"* — questions that span Teams, email, meetings, PRs, and documents simultaneously.

No single native client (Outlook, Teams, ADO) can do cross-source search. Brain already maintains pre-built JSON search indexes (`.search_index/`) covering ~3,900 entries across 7 source types. Surfacing this in AgentLink adds genuine value that native tools cannot match.

## User Experience

### Entry Point

**Search is the default Feed landing page.** When the user switches to Feed mode in the sidebar:

1. Sidebar shows Search as the **first tab** (before Recaps, Briefings, DevOps, Projects)
2. Main area renders `SearchFeed.vue` — a centered search box with source stats below
3. No sidebar sub-list needed (unlike Recaps/DevOps which show item lists) — the sidebar tab is just a navigation button

### Interaction Details

1. **Type in search box** → 300ms debounce → send `brain_search` message to agent
2. **Agent streams results** — sends results grouped by source type as they're found, allowing incremental rendering
3. **Results grouped by source** — Teams, Email, Meetings, PRs, Work Items, Documents — each section with a count badge
4. **Keyword highlighting** — search terms bold in `body_preview` snippets
5. **Click a result** → navigates to the relevant detail view if available:
   - PR/WI → switches to DevOps feed detail
   - Meeting → could link to recap if matched
   - Teams/Email → for now just expands inline (no separate detail page, since raw messages don't have AI-enriched detail views)
6. **Empty state** — source stats + recent search suggestions
7. **No results** — friendly message suggesting broader terms

## Technical Design

### Data Source

JSON search indexes at `~/BrainData/.search_index/`:

| File | Entries | Key Fields |
|------|---------|------------|
| `teams.json` | 1,984 | id, timestamp, sender, sender_email, chat, body_preview, mentioned_users, file |
| `emails.json` | 1,831 | id, timestamp, sender, sender_email, subject, folder, importance, thread_id, body_preview, file |
| `meetings.json` | 43 | id, timestamp, meeting_name, body_preview, file |
| `pull_requests.json` | 23 | id, pr_number, title, url, project, repository, status, total_mentions, file |
| `work_items.json` | 10 | id, title, url, project, total_mentions, file |
| `documents.json` | 10 | id, title, url, total_mentions, file |
| `aml_jobs.json` | 0 | (empty) |

Each file has structure: `{ source, version, generated, count, entries: [...] }`

### Agent Side — `agent/src/search.ts`

```typescript
export interface SearchRequest {
  query: string;
  sources?: string[];     // optional filter, e.g. ['teams', 'emails']
  limit?: number;         // max results per source (default 20)
}

export interface SearchResultEntry {
  id: string;
  source: string;         // 'teams' | 'emails' | 'meetings' | 'pull_requests' | 'work_items' | 'documents'
  timestamp: string;
  title: string;          // sender/subject/meeting_name/pr_title depending on source
  subtitle: string;       // chat name / folder / project
  snippet: string;        // body_preview with match context
  url?: string;           // web_link if available
  file?: string;          // relative path in BrainData
  extra?: Record<string, any>; // source-specific metadata (importance, pr_number, etc.)
}

export interface SearchResultGroup {
  source: string;
  label: string;          // "Teams", "Email", "Meetings", etc.
  count: number;          // total matches in this source
  entries: SearchResultEntry[];  // up to `limit` entries
}
```

**Search algorithm:**
1. On first `brain_search` request, load and cache all `*.json` index files in memory (~1-2MB total)
2. Case-insensitive substring match across searchable text fields per source type
3. Score by recency (newer first) as tie-breaker
4. Stream results back as `search_results` message with all groups at once (indexes are small enough that in-memory search completes in <50ms — no need for per-source streaming)

**Searchable fields per source:**

| Source | Fields searched |
|--------|----------------|
| teams | sender, chat, body_preview, mentioned_users |
| emails | sender, subject, body_preview |
| meetings | meeting_name, body_preview |
| pull_requests | title, pr_number, project, repository |
| work_items | title, project |
| documents | title |

**Cache invalidation:** Reload index files every 5 minutes (check `_meta.json` timestamp). Or simply reload on each `brain_search` request if performance is acceptable (files are small).

### Protocol — WebSocket Messages

**Web → Agent:**

| Type | Fields | Purpose |
|------|--------|---------|
| `brain_search` | `query: string`, `sources?: string[]`, `limit?: number` | Trigger search |

**Agent → Web:**

| Type | Fields | Purpose |
|------|--------|---------|
| `search_results` | `query: string`, `groups: SearchResultGroup[]`, `totalResults: number` | Search results |
| `search_index_stats` | `sources: { name, count, generated }[]` | Index metadata for empty state |

### Web Side

**New files:**

| File | Purpose |
|------|---------|
| `modules/search.js` | `createSearch(deps)` — search state, send/receive, debounce |
| `modules/handlers/search-handler.js` | `createSearchHandlers(deps)` — route WS messages to search module |
| `components/SearchFeed.vue` | Main search UI — search box, results grouped by source |
| `css/search.css` | Search-specific styles |

**`modules/search.js` state:**

```javascript
const searchQuery = ref('');
const searchResults = ref([]);       // SearchResultGroup[]
const totalResults = ref(0);
const searching = ref(false);
const indexStats = ref([]);          // from search_index_stats
const searchError = ref(null);
```

**`SearchFeed.vue` structure:**

```vue
<template>
  <!-- Search input -->
  <div class="search-input-container">
    <input v-model="searchQuery" placeholder="Search across all Brain data..."
           @input="debouncedSearch" autofocus />
  </div>

  <!-- Empty state: show source stats -->
  <div v-if="!searchQuery && indexStats.length" class="search-stats">
    <span v-for="s in indexStats">{{ s.icon }} {{ s.count }} {{ s.label }}</span>
  </div>

  <!-- Loading -->
  <div v-if="searching" class="search-loading">Searching...</div>

  <!-- Results grouped by source -->
  <div v-for="group in searchResults" class="search-group">
    <div class="search-group-header">
      <span class="search-group-icon">{{ group.icon }}</span>
      <span class="search-group-label">{{ group.label }}</span>
      <span class="search-group-count">{{ group.count }}</span>
    </div>
    <div v-for="entry in group.entries" class="search-result-card" @click="openResult(entry)">
      <div class="search-result-title" v-html="highlight(entry.title)"></div>
      <div class="search-result-subtitle">{{ entry.subtitle }}</div>
      <div class="search-result-snippet" v-html="highlight(entry.snippet)"></div>
      <div class="search-result-meta">{{ formatDate(entry.timestamp) }}</div>
    </div>
  </div>

  <!-- No results -->
  <div v-if="searchQuery && !searching && totalResults === 0" class="search-no-results">
    No results found for "{{ searchQuery }}"
  </div>
</template>
```

**Keyword highlighting:** Simple regex-based bold wrapping (escape HTML first, then wrap matches with `<mark>`). Same approach as standard search UIs.

### Integration Points

**store.js:**
- Add `_search` module via `createSearch(deps)`
- Provide via `provide('search', store._search)`
- Set `currentView` default to `'search-feed'` when entering Feed mode

**connection.js:**
- Add `search-handler.js` to handler dispatch
- Add `search` to `handlerDeps` late-binding (if needed; likely not needed since search is stateless)

**Sidebar.vue:**
- Add Search button as first item in `.feed-sidebar-nav`
- No sub-list component needed for search

**i18n (en.json / zh.json):**
- `search.placeholder`, `search.noResults`, `search.searching`, `search.stats`, source labels


**Behaviors:**
- Each source group shows first 3 results, with "Show N more" to expand
- Search terms highlighted with `<mark>` bold
- Email cards show importance badge (High = warning color)
- PR/WI cards clickable → navigate to DevOps Detail view
- Results sorted by timestamp (newest first) within each group

## Scope & Non-Goals

**In scope:**
- Search tab as default Feed entry
- Cross-source keyword search via JSON indexes
- Results grouped by source with keyword highlighting
- Click-to-navigate for PR/WI (to existing DevOps detail)

**Out of scope (future):**
- Full-text search via SQLite FTS (Option B) — deferred until JSON search proves insufficient
- Filters (date range, source type checkboxes) — can add later, start simple
- Search history / saved searches
- Click-to-detail for Teams/Email (no AI-enriched view exists for raw messages)
- Brain Mode "ask about this result" chat integration
